# Android 设备兼容矩阵

## 1. 文档目标

这份文档用于定义 TacticBoard 在 Android 第一阶段的设备兼容性目标、验证优先级和不承诺范围。

目标：

- 把“兼容哪些安卓设备”从口头判断变成明确文档
- 区分 `必须覆盖`、`优先验证`、`仅观察`、`暂不承诺` 四类范围
- 为 Android 第一阶段技术验证、验收和后续对外说明提供统一口径

一句话目标：

- `先冻结 Android 第一阶段的兼容范围，再开始讨论“广泛机型兼容”。`

## 2. 当前结论

当前这套 Android 方案对机型兼容性的判断应写成：

- 现阶段已经有较好的跨机型工程基础
- 但还不能宣称“兼容各个安卓机型”
- Android 第一阶段应只承诺覆盖 `主流手机 + 主流平板` 的核心主链路

更准确的对外表述是：

- `当前 Android 第一阶段面向主流安卓手机与平板进行技术验证和主链路适配，暂不承诺覆盖所有品牌、ROM 和设备形态。`

## 3. 使用规则

这份矩阵用于：

- Android 技术验证阶段
- Android 第一阶段验收阶段
- Android 对外口径收口阶段

这份矩阵不用于：

- 证明当前已经完成 Android 全量适配
- 直接定义应用商店最终支持列表
- 代替 `android-acceptance-standard.md`

关系说明：

- `android-acceptance-standard.md` 决定“什么叫 Android 第一阶段完成”
- 本文决定“这些完成条件至少要在哪些设备等级上验证”

配套执行文档：

- `android-device-validation-plan.md`

配套记录模板：

- `../DocsReview/android-device-compatibility-validation-template.md`

### 3.1 兼容性口径的证据等级

本文里的“P0 已覆盖”或“第一阶段可宣称完成”默认建立在：

- `android-device-validation-plan.md` 已按要求执行
- P0 设备上的关键能力都有对应验证记录
- 系统分享、系统文件选择器、本地复制、生命周期项至少具备一项设备侧硬证据
- 结论来自已提交、可追踪的实现范围，而不是仅本地实验

不得用以下内容扩大兼容承诺：

- 未提交本地实验
- 只改 `src-tauri/gen/**`、`src-tauri/target/**` 或 `src-tauri/vendor/**` 的临时补丁
- 只有自动化或烟雾证据、没有设备侧硬证据的系统集成项

## 4. 兼容性分级

### 4.1 P0：第一阶段必须覆盖

定义：

- 不通过就不能宣称 Android 第一阶段完成

### 4.2 P1：第一阶段强烈建议覆盖

定义：

- 不通过不一定阻塞 Android 第一阶段
- 但必须留下明确风险记录与后续处理计划

### 4.3 P2：第一阶段观察项

定义：

- 可以记录现象
- 不作为 Android 第一阶段完成门槛

### 4.4 Out of Scope：当前不承诺

定义：

- 当前阶段不纳入完成条件
- 也不应在对外口径中默认写成已支持

## 5. 当前不冻结的内容

在 Android 壳层和运行时验证真正完成之前，当前不建议把以下内容写死成正式公开承诺：

- 最低支持 Android 版本号
- 最低内存配置
- 最低 CPU / GPU 性能线
- 所有 ROM 厂商覆盖承诺
- 所有屏幕形态覆盖承诺

原因：

- 当前 Android 仍处于第一阶段预实现 / 预验证状态
- 如果过早写死最低线，后面最容易出现“文档先承诺，工程再补救”的问题

因此当前矩阵优先冻结的是：

- 设备类别
- 设备优先级
- 必须验证的能力
- 明确不承诺的范围

## 6. Android 第一阶段设备兼容矩阵

| 设备类别 | 当前定位 | 优先级 | 第一阶段要求 |
|---|---|---:|---|
| 主流手机（竖屏） | 第一阶段核心目标设备 | P0 | 必须完成启动、主链路、PNG 导出、系统分享、素材导入、保存与恢复 |
| 主流手机（横屏） | 第一阶段必须覆盖的方向切换场景 | P0 | 必须证明横竖屏切换后主链路不断裂 |
| 主流平板 | 第一阶段核心目标设备 | P0 | 必须完成编辑主链路，并验证画布、工具、属性、步骤入口可达 |
| 入门级 / 低性能手机 | 第一阶段风险观察设备 | P1 | 建议验证启动速度、编辑响应、导入与导出是否出现明显阻塞 |
| 深度定制 ROM 手机 | 第一阶段风险观察设备 | P1 | 建议验证系统分享、文件选择器、后台恢复是否与标准系统有明显差异 |
| 厂商激进后台管理设备 | 第一阶段风险观察设备 | P1 | 建议重点观察切后台 / 回前台、自动保存、上下文保持 |
| 折叠屏设备 | 第一阶段观察项 | P2 | 可记录现象，但不作为完成门槛 |
| 分屏 / 小窗 / 桌面模式 | 第一阶段观察项 | P2 | 可记录现象，但不作为完成门槛 |
| 外接键鼠 / 触控笔 | 第一阶段观察项 | P2 | 可记录现象，但不作为完成门槛 |
| ChromeOS / Android TV / 车机 | 当前不承诺 | Out of Scope | 不纳入第一阶段完成条件 |

## 7. P0 设备必须验证的能力

如果要宣称 Android 第一阶段完成，P0 设备上至少必须验证这些能力：

### 7.1 启动与进入

- 应用可启动
- 工作台可进入
- 项目页 / 设置页 / 编辑器入口可进入

### 7.2 核心主链路

- 新建项目或打开项目
- 进入编辑器
- 完成至少一种核心对象编辑
- 保存
- 返回工作台或项目页
- 再次打开项目

### 7.3 导出与分享

- PNG 导出成功
- 系统分享链路可进入
- 用户取消分享时不报错
- 至少留下一项设备侧硬证据，证明分享链路真实命中

### 7.4 素材导入

- 系统文件选择器可打开
- 球员头像导入可用
- 参考底图导入可用
- 导入后已完成本地复制
- 至少留下一项设备侧硬证据，证明系统文件选择器与本地复制真实命中

### 7.5 生命周期与方向切换

- 切后台再回前台，不明显丢上下文
- 横竖屏切换后主链路不断裂
- 至少留下一项明确手动记录，标明验证设备与切换场景

## 8. P1 设备重点观察项

P1 设备不一定阻塞第一阶段，但必须重点观察：

- 启动是否明显过慢
- 编辑器是否明显掉帧或操作不稳定
- 系统分享是否出现 ROM 级差异
- 系统文件选择器是否出现 ROM 级差异
- 切后台 / 回前台后是否更容易丢状态
- 自动保存在低性能或高回收环境下是否更容易失败

如果 P1 设备上出现问题，验收结论必须至少写明：

- 具体设备类别
- 具体现象
- 是否影响 P0 主流设备完成结论
- 是否作为已知限制保留到下一阶段

## 9. P2 设备与形态观察项

P2 设备和形态当前只做观察，不作为 Android 第一阶段完成条件：

- 折叠屏展开 / 折叠态切换
- 分屏 / 小窗模式
- 桌面模式 / Dex 类场景
- 外接键盘 / 鼠标 / 触控笔

要求：

- 可以记录问题
- 不能把 P2 现象直接上升为 P0 阻塞
- 也不能在对外说明中默认写成已经支持

## 10. 暂不承诺范围

当前阶段不应承诺这些兼容范围：

- “兼容所有安卓品牌手机”
- “兼容所有安卓版本”
- “兼容所有国产 ROM 行为差异”
- “兼容折叠屏全部姿态”
- “兼容平板 / 手机 / 分屏 / 桌面模式所有布局”
- “兼容低端设备上的完整桌面等价体验”

如果后续需要对外扩大承诺，必须先补：

- 新的兼容矩阵结论
- 新的验收记录
- 必要的 UI / 生命周期 / 性能修正

## 11. 当前推荐验证设备组合

Android 第一阶段建议至少准备以下验证组合：

### 11.1 P0 最小组合

- 1 台主流安卓手机（竖屏主验证）
- 同一台手机完成横屏验证，或补 1 台常见横屏可用设备
- 1 台主流安卓平板

### 11.2 P1 建议补充组合

- 1 台入门级或性能偏弱设备
- 1 台深度定制 ROM 设备

### 11.3 当前不要求的组合

- 折叠屏专项验证
- ChromeOS 专项验证
- Android TV / 车机专项验证

## 12. 兼容性结论写法建议

### 12.1 当前可以写的说法

- `当前 Android 第一阶段面向主流安卓手机与平板进行技术验证。`
- `当前优先保证触控主链路、PNG 导出、系统分享、素材导入和生命周期稳定性。`
- `当前不承诺覆盖所有 ROM、所有屏幕形态和所有安卓设备等级。`
- `当前兼容结论只对已验证的 P0 设备等级成立，系统分享 / 文件选择器 / 生命周期条目已按要求留下设备侧硬证据。`

### 12.2 当前不应该写的说法

- `已兼容各个安卓机型`
- `已兼容所有安卓系统版本`
- `已完成广泛安卓设备适配`
- `已覆盖所有手机和平板场景`
- `只有自动化或本地实验，也可以算系统集成项已通过`

## 13. 与验收标准的关系

本文与 `android-acceptance-standard.md` 的关系如下：

- 验收标准决定：`什么叫完成`
- 兼容矩阵决定：`在哪些设备等级上完成才算数`

因此：

- 如果 P0 设备没通过，则不能宣称 Android 第一阶段完成
- 如果只有 P1 或 P2 设备有问题，不一定阻塞完成，但必须记录已知风险

## 14. 与开发指导的关系

本文与 `android-development-guide.md` 的关系如下：

- 开发指导决定：`怎么做`
- 兼容矩阵决定：`优先在哪些设备上做`

也就是说：

- 实施时优先围绕 P0 设备做问题闭环
- P1 设备用于风险发现
- P2 设备只做观察，不倒逼第一阶段范围膨胀

## 15. 当前结论

当前最合理的兼容性结论不是：

- `Android 方案已经能覆盖所有机型`

而是：

- `Android 第一阶段已经具备面向主流手机与平板建立兼容基线的文档基础。`

接下来的正确动作不是扩大承诺，而是：

1. 先在 P0 设备上完成技术验证
2. 再在 P1 设备上观察 ROM / 性能 / 生命周期差异
3. 最后再决定是否扩大公开兼容范围

在没有 P0 设备侧硬证据前，当前最多只能说“兼容范围文档已冻结”，不能说“兼容基线已验证完成”。

具体执行方式见：

- `docs/android-packaging/android-device-validation-plan.md`
