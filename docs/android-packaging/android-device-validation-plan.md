# Android 设备验证计划

## 1. 文档目标

这份文档用于把 `android-device-compatibility-matrix.md` 里的设备分级和兼容边界，转成 Android 第一阶段可执行的验证计划。

目标：

- 明确 Android 第一阶段设备验证应该怎么排期
- 明确 P0 / P1 / P2 设备各自承担什么验证职责
- 明确每一轮设备验证至少要留下什么证据
- 让后续技术验证不再停留在“建议测哪些设备”，而是能直接执行

一句话目标：

- `把兼容性边界变成设备验证执行顺序。`

## 2. 适用范围

本文件覆盖：

- Android 第一阶段的设备准备与验证顺序
- 不同设备等级的验证职责
- 每一轮设备验证的最小证据要求
- 设备问题如何判定为阻塞 / 风险 / 观察项

本文件不覆盖：

- 应用商店支持范围
- 公开市场发布设备白名单
- 机型适配实现细节
- Android 原生实现方式细节
- 第二阶段及之后的兼容性扩展

## 4. 核心原则

Android 第一阶段设备验证必须遵守：

1. 先跑 P0，再跑 P1，最后才看 P2
2. P0 决定“能不能宣称第一阶段完成”
3. P1 决定“已知风险是否可接受”
4. P2 只负责发现未来问题，不倒逼第一阶段范围膨胀
5. 没有记录的设备验证，默认视为未验证

### 4.1 证据等级与提交状态口径

设备验证默认区分三类证据：

- 自动化证据：`build / test / lint`、单元测试、集成测试
- 运行烟雾证据：应用能安装、能启动、页面能进入
- 设备侧硬证据：录屏 / 截图、运行时日志、导出或本地复制产物、明确可复现的手动验收记录

额外规则：

- Round 3 / 4 / 5 以及所有 P0 系统集成项，设备侧硬证据是默认要求
- 未提交本地实验不能直接写成某轮“已通过”
- 如果结论依赖 `src-tauri/gen/**`、`src-tauri/target/**` 或 `src-tauri/vendor/**` 临时补丁，必须明确写出该轮仍不构成第一阶段设备验证通过

## 5. 设备等级职责

### 5.1 P0 设备

职责：

- 作为 Android 第一阶段完成判定的核心设备集
- 必须承载完整主链路和完整验收关键场景

P0 不通过的后果：

- 不能宣称 Android 第一阶段完成

### 5.2 P1 设备

职责：

- 作为 ROM / 性能 / 生命周期风险发现设备
- 用于发现 P0 上不明显、但真实市场中高概率出现的问题

P1 不通过的后果：

- 不一定阻塞第一阶段完成
- 但必须留下明确风险记录和后续处理计划

### 5.3 P2 设备

职责：

- 观察未来形态兼容问题
- 为后续阶段扩展提供早期信号

P2 不通过的后果：

- 不阻塞第一阶段完成
- 只作为观察结论记录

## 6. 第一阶段建议设备池

### 6.1 P0 最小设备池

至少准备：

1. 主流安卓手机（竖屏主验证） 1 台
2. 主流安卓手机横屏验证路径 1 组
   - 可以是同一台手机切横屏
   - 也可以补 1 台横屏体验更稳定的设备
3. 主流安卓平板 1 台

### 6.2 P1 建议设备池

建议补：

1. 入门级 / 低性能安卓手机 1 台
2. 深度定制 ROM 手机 1 台
3. 激进后台管理倾向设备 1 台（如果与上面不是同一类）

### 6.3 P2 观察设备池

有条件再补：

1. 折叠屏设备
2. 支持分屏 / 小窗 / 桌面模式的设备
3. 外接键鼠 / 触控笔场景设备

## 7. 验证轮次顺序

### Round 0：环境确认

目标：

- 确认设备池已经准备好
- 确认每台设备的记录字段完整

至少记录：

- 设备名
- 设备类别
- P0 / P1 / P2 等级
- Android 版本
- 手机 / 平板
- ROM / 定制系统说明

### Round 1：P0 启动与页面进入

目标：

- 验证 Android 应用可以在 P0 设备上稳定启动并进入关键页面

必须覆盖：

- 工作台
- 项目页
- 设置页
- 编辑器入口

### Round 2：P0 主链路

目标：

- 验证新建 / 打开 -> 编辑 -> 保存 -> 返回 -> 再打开

### Round 3：P0 导出与分享

目标：

- 验证 PNG 导出与系统分享闭环

额外要求：

- 至少留下一项设备侧硬证据，证明系统分享面板或真实分享链路已命中

### Round 4：P0 素材导入

目标：

- 验证球员头像导入
- 验证参考底图导入
- 验证系统文件选择器 + 本地复制

额外要求：

- 至少留下一项设备侧硬证据，证明系统文件选择器与本地复制主路径已命中

### Round 5：P0 生命周期与方向切换

目标：

- 验证切后台 / 回前台
- 验证横竖屏切换
- 验证上下文稳定性

额外要求：

- 至少留下一项明确手动记录，标明真机或模拟器、切换步骤和结果

### Round 6：P1 风险发现

目标：

- 观察性能、ROM 差异、系统分享差异、文件选择器差异、后台恢复差异

### Round 7：P2 观察记录

目标：

- 记录折叠屏 / 分屏 / 桌面模式等现象
- 不将其上升为第一阶段完成门槛

## 8. 场景到设备等级的映射

| 场景 | P0 | P1 | P2 |
|---|---:|---:|---:|
| 启动与进入 | 必须 | 建议 | 观察 |
| 主链路 | 必须 | 建议 | 观察 |
| PNG 导出 | 必须 | 建议 | 观察 |
| 系统分享 | 必须 | 建议 | 观察 |
| 球员头像导入 | 必须 | 建议 | 观察 |
| 参考底图导入 | 必须 | 建议 | 观察 |
| 切后台 / 回前台 | 必须 | 建议 | 观察 |
| 横竖屏切换 | 必须 | 建议 | 观察 |
| 启动速度 / 低性能表现 | 可记录 | 必须观察 | 可不测 |
| ROM 特有差异 | 可记录 | 必须观察 | 可不测 |
| 折叠屏 / 分屏 / 桌面模式 | 不要求 | 可选 | 观察 |

## 9. 每一轮验证的证据要求

每一轮设备验证至少要留下：

- 本轮验证范围
- 本轮设备列表
- 每台设备对应等级（P0 / P1 / P2）
- 本轮运行命令
- 本轮手动场景
- 每台设备的通过 / 失败 / 仅观察结果
- 当前剩余风险
- 哪些条目只有自动化 / 烟雾证据，哪些条目已经具备设备侧硬证据
- 本轮结论依赖的是已提交改动，还是仅本地实验
- 是否涉及 `src-tauri/gen/**`、`src-tauri/target/**` 或 `src-tauri/vendor/**` 临时补丁

建议使用：

- `docs/DocsReview/android-technical-validation-template.md`
- `docs/DocsReview/android-device-compatibility-validation-template.md`

如果当前轮次属于 Android Phase 1 的某个切片，建议同时标注：

- 当前 Slice
- 当前 Slice Exit Decision

切片定义见：

- `docs/android-packaging/android-phase1-slice-plan.md`

## 10. 阻塞判定规则

### 10.1 阻塞 Android 第一阶段完成的情况

以下情况默认阻塞 Android 第一阶段完成：

- 任一 P0 设备无法完成主链路
- 任一 P0 设备无法完成 PNG 导出与系统分享闭环
- 任一 P0 设备无法完成两类素材导入
- 任一 P0 设备在生命周期或方向切换下主链路明显断裂
- 任一 P0 系统集成项只有自动化或烟雾证据，没有对应设备侧硬证据

### 10.2 不一定阻塞的情况

以下情况不一定阻塞 Android 第一阶段完成，但必须记录：

- P1 设备上出现明显性能问题
- P1 设备上出现 ROM 级差异
- P2 设备上出现折叠屏 / 分屏 / 桌面模式异常

### 10.3 不能默认忽略的情况

即使不阻塞，也必须留痕：

- 深度定制 ROM 差异
- 激进后台管理导致的恢复异常
- 系统分享面板行为差异
- 系统文件选择器行为差异

## 11. 已知限制记录规则

如果某台设备或某类设备存在问题，记录里至少要写：

- 设备类别
- 设备等级
- Android 版本
- 问题场景
- 现象描述
- 是否影响 P0 完成结论
- 后续处理建议

不允许的记录方式：

- 只写“某些机型有问题”
- 只写“安卓兼容性一般”
- 不区分是 P0 / P1 / P2 设备

## 12. 第一阶段设备验证退出条件

只有在以下条件同时满足时，才能认为 Android 第一阶段设备验证部分通过：

1. P0 设备池全部完成主链路验证
2. P0 设备池全部完成 PNG 导出与系统分享验证
3. P0 设备池全部完成球员头像导入与参考底图导入验证
4. P0 设备池全部完成生命周期与方向切换验证
5. 至少完成一轮 P1 风险观察
6. 已知限制被明确记录
7. P0 系统分享、系统文件选择器、本地复制、生命周期 / 方向切换都已留下对应设备侧硬证据
8. 退出结论不依赖未提交本地实验或生成目录 / vendor 临时补丁

P2 观察项不是退出条件。

## 13. 当前建议执行顺序

建议按这个顺序实际推进：

1. 先准备 P0 最小设备池
2. 用 P0 跑完整主链路
3. 用 P0 跑导出 / 分享 / 导入 / 生命周期
4. 再补 P1 设备做风险发现
5. 最后再决定是否补 P2 观察验证

不建议：

- 折叠屏先测
- 分屏 / Dex 先测
- 大量铺开机型但 P0 主链路还没跑通

## 14. 当前结论

兼容矩阵解决的是：

- `承诺哪些设备等级`

而这份验证计划解决的是：

- `先拿哪些设备做验证`
- `按什么顺序验证`
- `什么结果算阻塞`
- `什么结果只算已知风险`

因此 Android 第一阶段设备兼容工作，应该先按本文执行，再讨论是否扩大公开兼容范围。

没有设备侧硬证据的关键轮次，最多只能写成“已完成烟雾验证”，不能写成“设备验证完成”。
