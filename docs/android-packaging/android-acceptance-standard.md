# Android 第一阶段验收标准

## 1. 文档目标

这份文档定义 TacticBoard 在 Android 打包第一阶段的专项验收标准。

目标：

- 让 Android 打包验收从“理论上能做”提升到“当前阶段是否算完成”
- 统一功能验收、平台验收、触控验收和失败场景验收标准
- 把“什么算完成、怎么验、哪些不算首发承诺”固定下来
- 给后续技术验证、手动测试和实现收口提供明确清单

## 2. 验收范围

本文件覆盖：

- Android 壳层基础接入后的启动与页面进入
- Android 主链路可用性
- Android 触控下的核心编辑路径
- Android PNG 导出与系统分享
- Android 系统文件选择器与素材导入
- Android 本地保存、草稿恢复与生命周期稳定性
- Android 首发边界与非承诺范围

本文件不覆盖：

- Windows 打包验收
- Android 应用商店首发分发
- Android 签名与发布运营流程
- 自动更新
- 项目文件导入导出
- 跨设备迁移
- Android 端 GIF 导出作为首发默认承诺

## 3. 验收等级

### 3.1 可验证技术路线

满足：

- Android 工程可初始化
- Android 应用可启动
- 首页 / 项目页 / 设置页 / 编辑器入口可进入
- 平台桥主路径没有立即阻塞错误

### 3.2 可内部演示

满足：

- Android 主链路可跑通
- 触控下核心编辑操作可用
- PNG 导出可完成
- 系统分享链路可进入
- 素材导入链路可进入并完成本地复制
- 主要失败分支有合理提示

### 3.3 可作为 Android 第一阶段完成

满足：

- Android 端完整主链路达到可演示与可复测状态
- 已明确首发不做的能力
- 已有明确的安装、运行与已知限制说明
- 没有因为 Android 接入破坏 Web 与 Windows 当前基线

### 3.4 第一阶段完成的设备覆盖边界

这里的“Android 第一阶段完成”默认指：

- 已在 `android-device-compatibility-matrix.md` 定义的 P0 设备等级上完成关键验证
- 也就是主流安卓手机与主流安卓平板的核心主链路成立

这里不默认等于：

- 已广泛覆盖所有安卓机型
- 已覆盖所有 ROM
- 已覆盖所有设备形态

因此：

- P0 设备通过，是第一阶段完成判定的硬条件
- P1 设备主要用于风险发现与记录
- P2 设备主要用于观察，不作为第一阶段完成门槛

## 4. 环境与前置验证

必须明确：

- 当前 Android 壳层所需工具链已准备完成
- 当前 Android 运行验证环境已准备完成
- 当前 Android 实际使用的初始化、运行、构建命令已记录

通过标准：

- Android 技术验证不因缺少基础工具链直接失败
- 验收记录中必须明确写出本轮实际使用的命令与设备环境

说明：

- 当前阶段不强制在本文件里冻结最终命令名
- 因为 Android 壳层尚未正式落地完成，验收时应以当轮技术验证记录中的实际命令为准

## 5. Web 回归验收

Android 接入不能以破坏现有 Web 版本为代价。

必须通过：

- `npm run dev` 仍可正常启动
- 浏览器环境下工作台、项目页、设置页、编辑器仍可访问
- 浏览器环境下主链路 `工作台 -> 新建/打开 -> 编辑 -> 保存 -> 返回 -> 继续编辑` 仍可走通
- 浏览器环境下 PNG 导出仍可工作
- 浏览器环境下现有素材导入路径仍可工作

失败判定：

- 为了接入 Android 而导致 Web 路由失效
- 为了接入系统分享而破坏浏览器导出
- 为了接入 Android 文件选择而破坏浏览器 `File` 导入路径

## 6. Windows 基线回归观察项

Android 接入阶段不要求每轮都重新完成完整 Windows 首发验收，但至少不能破坏当前 Windows 文档已经冻结的边界。

必须确认：

- 当前平台桥没有把 Windows 分支语义改坏
- `windows-tauri` 与 Android 分支的判断逻辑没有混淆
- Android 接入没有把 Windows 原生保存 / 原生导入的边界重新写回组件层

失败判定：

- Android 接入导致 Windows 平台判断失效
- Android 接入导致 Windows 端导出 / 导入语义被改成不兼容状态

## 7. 启动与路由验收

必须通过：

- Android 应用可启动
- 工作台可打开
- 项目页可打开
- 设置页可打开
- 编辑器入口可打开
- 从工作台 / 项目页进入编辑器主路径可用
- 页面切换不会立即进入空白页或死路

要求：

- Android 端继续沿用当前工作台 / 项目页 / 编辑器页 / 设置页体系
- 不因为 Android 接入而复制第二套页面树

## 8. Android 主链路验收

必须通过以下完整路径：

1. 打开工作台
2. 新建项目或打开已有项目
3. 进入编辑器
4. 完成至少一种核心对象编辑
5. 切换步骤或播放步骤
6. 触发保存或等待自动保存
7. 返回工作台或项目页
8. 再次打开项目
9. 主要内容仍存在

失败判定：

- 保存后项目消失
- 重开后主要内容明显丢失
- 返回工作台后主链路中断
- Android 端只能启动，但无法完成“编辑 -> 保存 -> 返回 -> 再打开”闭环

## 9. 触控编辑验收

必须通过：

- 触控下可选择对象
- 触控下可移动对象
- 触控下可切换步骤
- 触控下可进入保存与导出主动作
- 触控下关键面板、抽屉或浮层可以正常打开和关闭

要求：

- Android 第一阶段首先保证主链路可用，不要求达到 Windows 桌面效率
- 手机端优先保证画布、工具、属性、步骤的可达性
- 平板端优先保证完整编辑能力，不强求桌面级多区布局完全等价

失败判定：

- 主要编辑动作只能依赖桌面鼠标行为才能完成
- 抽屉 / 面板打开后遮挡主路径且无法返回
- 关键按钮点击区域不足，导致主链路无法稳定操作

## 10. PNG 导出与系统分享验收

### 10.1 PNG 导出

必须通过：

- 可触发 PNG 导出
- PNG 二进制生成成功
- 导出失败时有明确提示
- 导出失败时不破坏当前编辑状态

### 10.2 系统分享

Android 第一阶段默认以“PNG 导出 + 系统分享”作为重点交付路径。

必须通过：

- 触发分享后能进入系统分享链路
- 用户取消分享时不报错
- 分享失败时有明确提示

### 10.3 第一阶段边界

必须明确：

- Android 第一阶段验收重点是 PNG 导出与系统分享闭环
- 如果当前实现同时支持“本地文件落地”，可以作为加分项记录
- 但不能把“本地落盘”写成 Android 第一阶段默认硬承诺，除非当轮实现已明确支持并完成验证

### 10.4 不算通过的情况

- 只能生成 PNG，但无法进入系统分享链路
- 分享失败后没有可区分提示
- 用户取消分享被当成错误

## 11. 素材导入验收

Android 第一阶段必须把素材导入规则说清楚，而不是只写“可以选文件”。

### 11.1 验收覆盖范围

本文件对 Android 素材导入至少覆盖两类场景：

- 球员头像导入
- 参考底图导入

阶段性技术验证允许某一轮实现先接通其中一类，但验收结论必须明确写出：

- 哪一类已经完成
- 哪一类仍未纳入本轮实现

不得把未实现的那一类默认写成已支持。

如果要判定“Android 第一阶段完成”，则默认要求：

- 球员头像导入已完成
- 参考底图导入已完成

只有在顶层产品 / 需求文档先明确收窄 Android 第一阶段范围时，才允许把其中一类从“第一阶段完成条件”中移出。

### 11.2 必须通过

- 可打开系统文件选择器
- 通过最小化授权完成图片选择
- 导入素材后复制到应用本地存储
- 导入完成后编辑器继续使用本地副本
- 用户取消选择时不报错
- 文件不合法时有提示
- 读取失败时有提示

### 11.3 失败判定

- 只能打开选择器但无法真正导入
- 长期依赖外部路径或临时授权引用
- 导入失败后留下空引用
- 组件层仍直接处理 Android 原生路径 / URI

## 12. 保存、草稿恢复与生命周期验收

### 12.1 本地保存

必须通过：

- 手动保存当前项目可用
- 自动保存草稿可用
- 最近项目继续编辑可用

### 12.2 草稿恢复

必须通过：

- 应用异常退出或中断后，下次进入时可恢复最近一次草稿或跳过
- 自动保存失败时保留上一份成功草稿

### 12.3 生命周期稳定性

必须验证：

- 切后台再回前台，不应明显丢失当前编辑上下文
- 横竖屏切换后，主链路不应立即断裂
- 导入素材后的本地引用在本轮运行内保持稳定

### 12.4 必须明确

- Android 第一阶段不承诺与 Web / Windows 自动数据互通
- Android 第一阶段不承诺跨设备迁移
- 卸载应用、清理数据或更换设备导致本地数据丢失，不视为第一阶段 bug，但必须有清晰提示策略

## 13. 响应式、方向与设备等级验收

必须覆盖：

- 手机竖屏主链路
- 手机横屏查看与编辑基本路径
- 平板主要编辑路径

必须通过：

- 画布优先可见
- 工具、属性和步骤入口在对应设备等级下可达
- 横竖屏切换后仍可继续主路径

失败判定：

- 某一主要设备等级下无法进入编辑主链路
- 横竖屏切换导致关键入口消失
- 平板布局与手机布局都不能稳定承接核心操作

## 14. 失败场景验收

必须覆盖：

- 用户取消分享
- 用户取消文件保存
- 用户取消图片选择
- PNG 导出失败
- 文件读取失败
- 图片 MIME 不支持
- 图片体积过大
- 素材本地复制失败
- 自动保存失败
- 切后台 / 回前台
- 横竖屏切换

要求：

- 用户取消不应显示错误
- 失败提示要能区分原因
- 失败后仍可继续使用应用
- 失败后用户可立即重试

## 15. Android 第一阶段不算首发承诺的内容

以下内容即使未来要做，也不算 Android 第一阶段未完成：

- Android 端 GIF 导出
- 应用商店首发分发
- 自动更新
- 应用内静默更新
- 项目文件导入导出
- 本地备份包
- 跨设备迁移
- 重型快捷键体系
- 复杂批量操作
- 复杂图层 / 分组管理
- 打印模式与教练演示增强
- 球员职责卡、战术对比、MP4 等后续增强能力

这些如果要纳入，必须在后续阶段单独补方案、架构和验收标准。

## 16. 推荐验收命令与证据要求

当前阶段建议至少保留：

```bash
npm run build
npm run test
npm run lint
```

Android 专项命令：

- 以当轮技术验证记录中的实际命令为准

验收记录至少应写明：

- 本轮使用的 Android 初始化命令
- 本轮使用的 Android 运行命令
- 本轮使用的 Android 构建命令
- 测试设备或模拟器环境
- 本轮对应的设备等级（P0 / P1 / P2）
- 通过了哪些场景
- 哪些能力本轮明确未纳入

建议优先配合以下模板留痕：

- `docs/DocsReview/android-technical-validation-template.md`
- `docs/DocsReview/android-device-compatibility-validation-template.md`

## 17. 手动验收场景

### 场景 A：启动与页面进入

- 启动 Android 应用
- 进入工作台
- 打开项目页 / 设置页 / 编辑器入口

### 场景 B：主链路

- 新建项目或打开项目
- 编辑
- 保存
- 返回
- 再打开

### 场景 C：PNG 导出与分享

- 导出 PNG
- 进入系统分享链路
- 检查取消与失败分支

### 场景 D：素材导入

- 导入球员头像或参考底图
- 检查是否完成本地复制
- 检查失败分支

### 场景 E：生命周期

- 切后台
- 回前台
- 横竖屏切换
- 检查当前编辑状态

### 场景 F：回归

- 验证 Web 主链路未损坏
- 验证 Windows 平台桥语义未被 Android 接入污染

## 18. 验收结论模板

实施完成后，Android 第一阶段验收结论建议至少包含：

- 是否完成 Android 技术路线验证
- 是否通过启动与页面进入
- 是否通过 Android 主链路
- 是否通过 PNG 导出
- 是否通过系统分享
- 是否通过素材导入
- 是否通过保存与草稿恢复
- 是否通过生命周期验证
- 本轮明确未纳入的能力有哪些
- 还存在哪些已知限制

## 19. 当前结论

Android 第一阶段验收重点不是“能不能把应用装到手机上”，而是：

- 能不能稳定启动
- 能不能稳定进入主页面和编辑器
- 能不能在触控下完成核心编辑主链路
- 能不能稳定导出 PNG 并进入系统分享
- 能不能稳定导入素材并继续编辑
- 能不能在生命周期变化后尽量保持当前上下文

达到这些，才算 Android 第一阶段真正完成。
