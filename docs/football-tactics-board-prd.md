# 足球战术板跨端本地应用 PRD

最后更新：2026-03-31
状态：正式版
关联文档：`docs/football-tactics-board-requirements.md`

## 1. 产品概述

产品定位：
一个面向业余足球爱好者、业余球队组织者和普通球迷的跨端本地战术板应用，支持在 Windows 与 Android 设备安装运行，用来快速制作阵型图、战术讲解图和回合复盘图，并通过本地保存、图片导出和 Windows 端基础 GIF 导出完成日常使用。内容创作者场景作为兼容方向保留，但不是第一阶段主目标。

产品关键词：
- 本地优先
- Windows + Android
- 免注册即可使用
- 足球专用
- 可视化战术编辑
- 长期资产沉淀

## 2. 产品目标

### 2.1 目标

- 提供明显优于现有工具的足球战术表达能力
- 让用户在 Windows 和 Android 上都能完成从新建到导出的完整创作闭环
- 降低使用门槛，让用户无需注册账号即可开始创作与保存
- 优先服务业余足球爱好者的高频表达需求，再逐步扩展专业能力和高级功能
- 让用户能保留最近项目和常用阵型，减少重复摆盘成本

### 2.2 非目标

当前版本明确不做：
- 在线账号体系
- 在线分享页 / 分享链接
- 完全自由绘制球场
- 训练器材对象
- 中立球员 / 占位对象
- 实时协作

## 3. 目标用户

### 3.1 核心用户与次级用户

- 业余足球爱好者
- 业余球队队长、组织者、队友
- 普通球迷 / 战术爱好者
- 足球内容创作者 / 自媒体作为兼容用户

### 3.2 用户诉求

- 快速表达阵型、跑位、球路和战术意图
- 在手机和电脑上都能完成完整创作，而不是只能查看
- 输出清晰、专业、易理解的战术图
- 能快速导出图片，必要时导出 GIF，方便发给队友、朋友查看
- 能保留最近项目和常用阵型，不必每次从零开始
- 不需要注册账号，不依赖云端服务也能使用

## 4. 核心使用场景

### 4.1 队内讨论阵型和战术

用户需要快速选择场地、套用阵型、摆放球员、绘制跑位与传球，并通过导出图片发给队友查看。

### 4.2 做简单讲解图和队内分享图

用户需要在标准球场上快速制作结构清晰、风格统一、导出质量高的战术图，用于队内沟通、个人记录和简单讲解。

### 4.3 赛后复盘某个回合

用户需要用多步骤帧还原回合、标记空间和压迫方向、补充文字说明，并通过播放、演示、导出图片或导出 GIF 展示完整过程。

## 5. 产品原则

- 第一阶段优先完成核心创作闭环，再逐步扩展专业能力
- 产品应保持完整产品感、专业感和视觉质量
- Windows 与 Android 首发，两端都支持完整编辑，但不要求所有专业效率功能完全一致
- Windows 端后续可优先承载快捷键、批量操作、复杂图层等重型效率能力
- Android 端优先保证触控场景下的新建、编辑、保存、导出完整可用
- 核心编辑、保存和导出能力默认离线可用，不依赖网络远程拉取
- 第一阶段不引入远程模板、远程主题、远程素材拉取等联网依赖
- 功能应优先服务足球表达，而不是泛化成通用白板
- 保存与编辑不应误覆盖用户已有内容
- 应用应易安装、易分发、易迁移，尽量降低首次使用门槛

## 6. 功能范围

### 6.1 场地与阵型系统

#### 场地模板

支持：
- 11 人制全场 / 半场
- 8 人制全场 / 半场
- 7 人制全场 / 半场
- 5 人制全场 / 半场

轻量场地自定义支持：
- 横版 / 竖版切换
- 显示 / 隐藏部分线条
- 首版提供 2 套球场视觉风格：简洁平面 / 草坪写实

不支持：
- 完全自由绘制球场

#### 阵型模板

11 人制：
- 首版内置阵型清单：4-3-3、4-2-3-1、4-4-2、4-4-2 菱形中场、4-5-1、4-1-4-1、4-3-1-2、3-5-2、3-4-3、3-4-2-1、3-4-1-2、5-3-2、5-4-1

8 人制：
- 首版内置阵型：3-3-1、3-2-2、2-3-2、4-2-1

7 人制：
- 首版内置阵型：2-3-1、3-2-1、2-1-2-1

5 人制：
- 首版内置阵型：2-2、1-2-1、2-1-1、1-1-2

通用能力：
- 选中阵型后可拖动微调
- 调整后的站位可保存为自定义阵型模板
制式切换规则：
- 从一种制式切换到不兼容的另一种制式时，系统必须先提示，不自动静默处理
- 切换时只提供：取消切换 / 仅切换球场制式并保留当前对象内容（当前阵型状态改为“自定义”）/ 按新制式重新生成阵型
- 第一版不做自动删除多余球员、不做智能重排、不做自动映射到新制式

- 阵型模板只包含站位结构，不包含文本、线路、步骤
- 站位类内容保存为阵型模板；后续若纳入项目模板，整张战术板可保存为项目模板
- 后续若新增阵型，以单独维护的内置阵型清单为准

### 6.2 编辑对象系统

#### 基础对象

- 球员
- 足球
- 文本说明
- 区域标记

#### 球员对象

支持：
- 颜色
- 号码
- 名字或简称
- 头像
- 用户上传真实头像
- 头像裁切和缩放
- 球衣卡片式支持显示已上传头像
- 位置标签
- 位置标签支持英文缩写和中文
- 队伍归属
- 首版球员展示样式仅支持简洁圆点式、球衣卡片式

不支持：
- 独立中立球员 / 占位对象
- 训练器材对象

#### 足球对象

第一版确认：
- 足球对象支持拖拽移动
- 足球对象支持两种基础样式：经典足球样式 / 简洁圆点样式
- 足球对象支持大小调整
- 足球对象支持与球员对象吸附
- 足球对象不支持名字、号码、头像等人物属性
- 第一版只支持单个足球对象

#### 文本说明对象

可用于：
- 标题
- 注释
- 简短说明
- 战术解释

- 文本仅支持 标题 / 正文说明 / 强调 三种预设样式
- 支持字号、颜色、粗细、对齐
- 支持无背景 / 纯色底
- 支持基础边框和圆角
- 支持透明度
- 第一版不支持富文本、段落级复杂排版、旋转文字

#### 区域标记对象

可用于：
- 区域高亮
- 区域框选
- 半透明遮罩
- 危险区域
- 压迫区域
- 关键空间
- 支持矩形区域
- 支持圆形 / 椭圆区域
- 第一版不支持自由多边形区域
- 区域标记统一支持边框、填充色、透明度

#### 参考底图

支持：
- 上传图片作为参考底图
- 参考底图作为项目级对象，默认跨全部步骤共享
- 参考底图缩放、拖动、裁切
- 参考底图透明度调节
- 参考底图锁定、隐藏、显示切换

不支持：
- 直接视频导入并截帧

目标：
- 提升真实回合复盘准确性
- 提升基于截图的内容讲解效率
- 保持产品边界在图片参考层，不扩展到视频编辑器

### 6.3 线路语义系统

支持：
- 跑位箭头
- 传球线路
- 带球线路
- 射门线路
- 防守压迫方向

要求：
- 第一版支持直线
- 第一版支持折线
- 第一版不支持曲线 / 贝塞尔曲线
- 第一版不支持弧线
- 线路起点和终点支持自动吸附到球员 / 足球对象
- 不同线路类型默认具备不同视觉样式
- 在线条颜色、粗细、虚实等层面可直接区分语义

### 6.4 步骤演示系统

支持：
- 多步骤帧
- 复制上一帧继续编辑
- 步骤播放
- 上一帧残影显示

规则：
- 复制上一帧时，球员、足球、文本、区域、线路对象继承同一对象身份
- 相邻步骤帧通过对象身份建立对应关系，用于残影和步骤连续性表达
- 在某一帧删除对象，不影响更早的历史帧
- 如果后续帧从当前帧复制而来，则被删除对象默认继续不存在
- 用户手动重新新增的对象视为新对象身份，不自动认作旧对象

目标：
- 支持战术讲解
- 支持回合复盘
- 支持分步骤内容展示

#### 步骤信息增强

支持：
- 步骤标题
- 步骤说明 / 讲解备注
- 步骤时间标签 / 回合标签
- 步骤缩略导航
- 导出时可选择带上步骤信息

目标：
- 让多步骤内容更清晰地表达“每一步在做什么”
- 支持复盘场景中的时间和回合语义
- 提升内容创作与教练讲解的结构化程度

### 6.5 图片与 GIF 导出

支持：
- PNG 导出
- GIF 导出（Windows 首发优先）
- 多种导出比例
- 高清导出
- Android 端通过系统分享发送导出图片

内容输出增强：
- PNG 图片支持透明背景导出
- 导出仅球员和线路的内容组合
- 标题区
- 图片导出比例沿用：16:9 / 1:1 / 9:16
- 标题 / 步骤说明 / 比赛信息统一按开关选择是否带上
- 第一阶段不做品牌区
- 第一阶段不做社媒导出安全区
- 第一阶段不做 MP4 导出
- 第一阶段不做在线分享链接

GIF 首发约束：
- GIF 仅基于步骤播放结果导出，不做镜头动画、音频和逐帧精修
- GIF 首发只提供 慢 / 标准 / 快 三档速度
- GIF 首发总时长上限为 15 秒，超出时提示减少步骤数量或选择更快的播放速度
- Windows 端将 GIF 导出纳入第一阶段；Android 端首版不强制支持 GIF 导出，优先保证 PNG 导出稳定

导出安全规则：
- 导出失败不得破坏当前项目内容或打断当前编辑状态
- 当导出尺寸、清晰度或 GIF 时长超出设备能力时，系统必须明确提示用户降级参数，而不是无反馈失败

### 6.6 比赛信息与项目元数据

支持：
- 基础比赛信息
- 比分、分钟、回合阶段
- 比赛信息展示在标题区、项目预览和导出图中

不支持：
- 比赛信息参与搜索和筛选
- 训练场景专用元数据

目标：
- 提升内容成品感
- 提升复盘场景的信息完整度
- 保持项目元数据边界清晰，不过度扩张到复杂检索体系

### 6.7 本地保存系统

支持：
- 无需注册账号，打开应用后可直接开始编辑
- 本地自动保存草稿
- 手动保存当前项目
- 最近项目列表
- 最近项目继续编辑
- 自定义阵型模板保存在本地

第一阶段不纳入：
- 项目文件导入导出
- 本地备份包
- 历史版本与本地快照
- 完整本地资产库

不支持：
- 邮箱 / 微信登录
- 云端保存
- 在线分享链接
- 团队空间、邀请码、资产权限控制

规则：
- 首次打开不强制注册或联网
- 核心编辑、保存、PNG 图片导出，以及 Windows 端 GIF 导出在无网络条件下必须可正常使用
- 应用启动、进入编辑器和打开最近项目不得被联网状态检查、远程拉取或更新检查阻塞
- 本地自动保存优先覆盖当前工作副本，不污染用户已手动保存的项目状态
- 应用异常退出后，下次启动优先提示恢复最近一次自动保存内容
- 自动保存与手动保存优先保障当前项目连续可编辑，而不是形成复杂文件管理体系
- 项目复制作为第一阶段替代版本管理的轻量方案
- 删除本地项目之前必须二次确认
- 第一阶段不保证跨设备迁移；卸载应用、清理应用数据或更换设备可能导致本地项目丢失，需在应用内明确提示用户
- 首阶段导入的球员头像和参考底图必须复制到应用本地存储，不得长期依赖外部文件路径或临时授权引用
- 自动保存写入失败时，必须保留上一份成功写入的本地草稿，不得把当前可恢复版本一并破坏
- 头像或参考底图导入失败时，不得在项目里留下不可用的空引用，必须明确提示用户重新选择素材

后续扩展约束：
- 若后续纳入项目文件或本地备份包，必须内嵌球员头像、参考底图等本地素材，不允许只记录设备本地路径
- 若后续纳入项目文件导入导出，旧版本文件可迁移打开，高于当前版本的文件需明确提示兼容风险
- 若后续纳入外部文件导入，导入同名文件时默认保留原文件，并为新导入副本生成可区分名称

## 7. 资产沉淀增强（后续阶段）

当前阶段定位：
- 除最近项目和自定义阵型模板外，本章能力不纳入第一阶段必做范围
- 第一阶段不要求完整实现项目模板、球队模板、阵容预设、球员资料库和历史版本
- 本章保留为后续扩展方向，避免后续讨论时重新从零定义

### 7.1 项目模板

后续版本方向：
- 保存为项目模板
- 项目模板包含步骤结构
- 项目模板包含默认视觉风格
- 项目模板包含文本结构
- 项目模板里可以继续套用阵型模板
- 项目模板可绑定球队模板或阵容预设
- 基于项目模板新建后默认不覆盖原模板

### 7.2 球队模板

后续版本方向：
- 保存为球队模板
- 球队模板包含球队名、队徽、主客场配色等基础信息
- 球队模板包含主场 / 客场两套视觉配置
- 一个用户可管理多支球队模板
- 球队模板可绑定默认阵容预设和默认阵型
- 加载球队模板后可局部改动，默认不覆盖原模板

### 7.3 阵容预设

后续版本方向：
- 保存球员名单或阵容预设
- 阵容预设保存“人员 + 站位”
- 阵容预设包含首发和替补
- 阵容预设包含位置与站位
- 同一支球队可保存多套阵容预设
- 阵容预设加载后可微调，默认不反向覆盖原预设

### 7.4 球员资料库

后续版本方向：
- 独立球员资料库
- 球员基础资料包含名字、头像、常用号码、常用位置
- 球员资料默认归属于某支球队
- 支持从球员资料库快速加入当前阵容

规则：
- 旧项目保留球员资料快照，不自动被后续修改覆盖
- 球员资料复制后完全独立，后续不自动同步
- 第一版不做“可能是同一个球员”的自动识别
- 再次复制同一球员资料时默认直接新建副本
- 第一版不支持手动合并球员资料
- 仅保留基础来源信息，例如来源项目、来源模板和复制时间

### 7.5 历史版本

后续版本方向：
- 项目副本
- 历史版本
- 自动快照 + 手动命名版本
- 自动快照按时间间隔触发
- 自动快照时间间隔为 10 分钟
- 仅在项目有改动时生成自动快照
- 每个项目最多保留 5 个自动快照
- 手动命名版本不受自动快照数量影响
- 超过上限时删除最旧的自动快照
- 恢复旧版本前先保留当前状态
- 恢复旧版本后自动生成一个新版本
- 手动命名版本支持标题 + 简短备注

### 7.6 依赖与快照规则

规则：
- 项目从项目模板、球队模板、阵容预设创建时，导入完整快照
- 项目模板绑定球队模板或阵容预设时，也保存一份快照
- 下游资产保留来源信息，例如来源于哪个项目模板、球队模板、阵容预设
- 上游资产更新后，不会自动影响已创建的项目或项目模板
- 支持手动从最新来源重新导入或刷新，但仅手动触发
- 上游资产被删除、移动、复制后，下游资产继续正常可用，但标记来源已变化
- 手动从最新来源重新导入 / 刷新前必须先提示用户
- 刷新冲突时只提供：取消 / 另存为新项目后刷新 / 直接刷新并覆盖当前项目
- 直接刷新并覆盖当前项目之前，系统自动留一个版本
- 支持另存为新项目后刷新
- 第一版不做局部合并和字段级冲突处理

### 7.7 新建项目覆盖顺序规则

规则：
- 新建项目时按“项目模板 -> 球队模板 -> 阵容预设 -> 用户最后手动选择”的顺序覆盖
- 如果项目模板里已经带了球队模板快照，优先使用项目模板里的快照，不再自动套用球队模板最新内容
- 如果球队模板绑定了默认阵容预设，而用户本次手动选择了其他阵容预设，则手动选择覆盖默认阵容预设
- 如果阵容预设已经定义站位，而用户本次又手动选择了阵型模板，则手动选择的阵型模板覆盖原站位
- 如果用户没有额外手动选择，则自动继续向下套用球队默认阵容预设和默认阵型
- 新建完成后落成一份最终快照，后续不再继续套用来源变化

目标：
- 让新建项目的来源优先级稳定且可预期
- 避免项目模板、球队模板、阵容预设、阵型模板同时存在时相互打架
- 保持最终项目仍然是快照型资产，而不是半动态引用

## 16. 关键产品模型

建议的核心实体（第一阶段）：
- Project：具体战术板项目
- StepFrame：项目中的单个步骤帧
- FormationTemplate：内置或自定义阵型模板
- MatchMeta：标题区与比赛信息元数据
- ExportConfig：导出时的临时配置，描述格式、尺寸比例与覆盖信息
- LocalDraft：本地自动保存草稿
- ProjectIndexEntry：项目列表与最近项目索引条目

可能的后续扩展实体：
- ProjectFile：项目文件，用于跨设备流转
- BackupPackage：本地备份包，用于迁移和恢复核心资产
- LocalLibrary：本地资产库，承载模板、球队、阵容预设和球员资料
- ProjectTemplate：项目模板
- TeamTemplate：球队模板
- LineupPreset：阵容预设
- PlayerProfile：球员资料
- ProjectVersion：项目版本 / 快照
