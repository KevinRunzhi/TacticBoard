## 硬性冲突

### R4-01 路由表与信息架构不完全一致

来源对照：

| 文档 | 定义 |
|---|---|
| IA §3 路由表 | `/editor`（新建）、`/editor/:projectId`（打开既有项目） |
| FRD editor.md §1 | 页面路由写 `/editor`、`/editor/:projectId`，但"必备能力"里同时列了"新建项目"和"打开既有项目" |
| IA §5 页面树 | 新建项目 → `/editor`；最近项目 → `/editor/:projectId` |

冲突点：

- IA 把 `/editor`（无 projectId）定义为**新建项目**的落地页。
- 但 FRD editor.md "必备能力"中把"新建项目"和"打开既有项目"并列，没有区分哪个路由负责哪个动作。
- 如果开发只看 FRD，会认为 `/editor` 和 `/editor/:projectId` 都可以新建也可以打开，模糊了路由语义。

建议：

- 在 `03-functional/frd/editor.md` 中明确写出：
  - `/editor` → 新建空白项目
  - `/editor/:projectId` → 打开既有项目
- 并补充：访问 `/editor/:projectId` 但 projectId 不存在时的异常处理规则（editor.md §9 已有异常描述但未绑定路由）。

## 边界缺口

### R4-02 scope-v1 未提及视觉样式与主题能力

PRD §9.1 和 Requirements §16 都明确了"双球员展示样式、成套主题、换肤"是第一阶段纳入项，但 `01-product/scope-v1.md` §5"V1 功能范围-纳入"清单完全没有列出视觉样式相关能力。

影响：

- scope-v1 是最精简的"做什么/不做什么"文档。如果视觉样式在 scope 层缺席，后续排期或裁剪时可能被误认为"不纳入 V1"。
- PRD §13 的发布范围清单已经写了"双球员展示样式、主题、换肤、风格预设等视觉样式能力"。

建议：

- 在 `scope-v1.md` §5"纳入"部分补入：`双球员展示样式（简洁圆点 / 球衣卡片）、成套主题、主客队换肤、文本/线路/区域风格预设`。

### R4-03 scope-v1 未提及编辑效率能力

PRD §8.1 和 Requirements §15 都明确了"撤销/重做、多选、对齐/吸附、锁定/隐藏"是第一阶段纳入项，但 `scope-v1.md` §5 没有列出任何编辑效率能力。

影响：

- 同 R4-02，容易让团队在 scope-v1 这个最短文档上遗漏核心编辑质量能力。

建议：

- 在 `scope-v1.md` §5"纳入"部分补入：`撤销/重做、多选、对齐/吸附、锁定/隐藏`。

### R4-04 scope-v1 未提及比赛信息与项目元数据

PRD §6.6 和 Requirements §11 明确了"比赛信息（比分、分钟、回合阶段）、标题区、导出图中展示比赛信息"是第一阶段纳入的能力，但 scope-v1 完全未提及。

影响：

- 比赛信息是导出成品感的关键组成，如果 scope 层缺席，后续可能在排期时被低估或遗漏。

建议：

- 在 `scope-v1.md` §5"纳入"部分补入：`比赛信息（比分、分钟、回合阶段）、标题区`。

### R4-05 scope-v1 "不纳入"清单比 Requirements §17 少写了多项

Requirements §17 "第一阶段明确不纳入"列了 16 项。scope-v1 §5 "不纳入"列了 12 项，缺少：

- 打印增强
- 球员职责卡
- 战术对比模式
- 内容创作包装、社媒适配、品牌化导出
- AI 文案辅助

影响：

- 这些条目在 requirements 和 PRD 中都已排除，但 scope-v1 作为最精简的"做/不做"摘要如果漏了它们，可能再次引发讨论。

建议：

- 把 scope-v1 §5"不纳入"对齐 requirements §17 的完整清单，或至少补一句"完整不纳入清单见 requirements §17"的引用。

## 隐藏风险

### R4-06 新建项目流缺少"场地与阵型选择时机"的明确划界

user-flows.md §1 新建项目流写的是线性序列：工作台 → 新建 → 编辑器 → 选择场地与阵型 → 开始编辑 → 保存导出。但信息架构 §6.1 工作台里已经列出了"内置阵型快捷开始"入口，workspace.md §2 也列了"内置阵型快捷开始"。

风险：

- 如果用户从工作台"内置阵型快捷开始"进入编辑器，此时场地和阵型**已经**预设好了，user-flows 的线性序列里"选择场地与阵型"这一步就变成了可跳过项。
- 但 user-flows 没有描述这条快捷路径，开发可能只做"先进空白编辑器再选"一条链路，忽略"带阵型直接进入编辑器"的场景。

建议：

- 在 `user-flows.md` §1 新增一个分支路径：工作台 → 点击阵型快捷入口 → 带预设场地+阵型进入编辑器 → 直接编辑。
- 或者把"内置阵型快捷开始"作为新建项目流的变体明确记录。

### R4-07 editor FRD 未提及"上一帧残影"渲染规则

PRD §6.4 明确列出"上一帧残影显示"为第一阶段支持项。PRD §9.7 统一渲染规则也写了"上一帧残影默认不导出，手动开启才导出"。Requirements §10 也列了"上一帧残影显示"。

但 `03-functional/frd/editor.md` 没有任何地方提及"残影"：

- §2 必备能力没有列出残影。
- §6 导出规则列了隐藏对象不导出、参考底图默认不导出，但没有说残影的导出默认行为。

风险：

- 残影涉及两个边界问题：(1) 编辑器视图里是否默认显示残影；(2) 导出时残影是否默认包含。如果 FRD 不写，开发时大概率会遗漏导出时的残影开关。

建议：

- 在 `editor.md` §4 或单独小节中补入：
  - 步骤切换时，上一帧残影默认显示
  - 导出时残影默认不导出，需显式开启

### R4-08 editor FRD 缺少制式切换规则

PRD §6.1 和 Requirements §7 都详细写了制式切换规则（提示 → 三选一 → 不做自动删除/重排/映射），但 `03-functional/frd/editor.md` §4 "场地与阵型"只写了一句"切换不兼容制式时必须明确提示"，没有把三选一选项列出来，也没有写"不做自动删除/重排/映射"的约束。

风险：

- 编辑器是制式切换的唯一实施入口。如果 FRD 只写"提示"，后面开发者可能做出一个只有"确认/取消"的对话框，遗漏"保留当前对象并切换球场"和"按新制式重新生成阵型"两个关键选项。

建议：

- 在 `editor.md` §4 "场地与阵型"中把制式切换的三个选项和三条不做约束写全：
  - 取消切换
  - 保留当前对象内容，仅切换球场制式（当前阵型标记为"自定义"）
  - 按新制式重新生成阵型
  - 第一阶段不做自动删除多余球员、不做智能重排、不做自动映射

### R4-09 responsive-rules 缺少"全屏演示模式"的响应式描述

PRD §9.2 和 Requirements §13 都提到了"全屏演示模式"。信息架构 §9 平台差异策略也隐含着全屏演示的需要。但 `02-ux/responsive-rules.md` 只描述了工作台/项目页/设置页和编辑器页的响应式行为，完全没有写全屏演示模式在不同设备上的表现。

风险：

- 全屏演示模式在 Windows 上可能是 F11 式全屏，在 Android 上可能需要系统栏隐藏。如果 responsive-rules 不提，开发可能只做桌面端全屏或者漏做移动端适配。

建议：

- 在 `responsive-rules.md` 新增一节描述全屏演示模式在桌面端和移动端的基本行为预期。

## 覆盖遗漏

### R4-10 FRD 层完全没有覆盖"导出安全规则"中的降级提示

PRD §6.5 和 Requirements §12 都定义了详细的导出安全规则：

- 导出失败不得破坏当前项目内容或打断当前编辑状态
- 隐藏对象不得进入导出；素材缺失时必须提示
- 超出设备能力时必须提示降级参数

`editor.md` §6 导出规则写了前两条的精简版本，但完全没有覆盖"超出设备能力时必须提示降级参数"这条。

风险：

- GIF 导出在低端 Windows 设备或高步骤数项目上很容易超限。如果 FRD 层不写降级提示规则，开发可能只做"导出成功/导出失败"两态，忽略"引导用户调参"的中间态。

建议：

- 在 `editor.md` §6 导出规则中补入：
  - 当导出尺寸、清晰度或 GIF 时长超出设备能力时，系统必须明确提示用户降级参数（减少步骤数量、降低分辨率或选择更快的播放速度），而不是无反馈失败

### R4-11 FRD 层和 UX 层都没有覆盖"素材导入规则"

PRD §6.7 和 Requirements §8.1 / §8.5 都定义了关键的素材导入规则：

- 首阶段导入的头像和参考底图必须复制到应用本地存储
- 不得长期依赖外部文件路径或临时授权引用
- 素材导入失败时不得留下不可用的空引用
- Android 端使用系统文件选择器完成最小化授权

但 `03-functional/frd/editor.md` 完全没有提及素材导入行为。`02-ux/user-flows.md` 也没有描述头像上传或参考底图导入的用户流。

风险：

- 素材导入是 Android 端最容易出权限和数据完整性问题的环节。如果 FRD 和 UX 层都不覆盖，开发可能直接引用外部路径、不做复制、不处理导入失败。

建议：

- 在 `editor.md` 中新增素材导入规则小节，至少覆盖：复制到本地、失败处理、Android 文件选择器。
- 考虑在 `user-flows.md` 中新增"素材导入流"，描述头像上传和参考底图导入的基本路径和异常处理。

### R4-12 editor-layout 缺少导出入口的位置描述

`02-ux/editor-layout.md` 描述了桌面端五个区域和移动端的布局，但没有提及"导出"按钮的位置预期。`03-functional/frd/editor.md` §3 顶部工具栏写了"导出入口"，但 editor-layout 没有呼应。

风险：

- 导出是主链路终点动作。如果布局文档不提它的位置，UI 实现时可能把导出入口放在不稳定的位置（比如被属性面板遮挡或者移动端需要多步才能触达）。

建议：

- 在 `editor-layout.md` 桌面端和移动端布局中分别标注导出入口的推荐位置（例如：桌面端顶部工具栏右侧，移动端顶部工具栏右侧或底栏）。
