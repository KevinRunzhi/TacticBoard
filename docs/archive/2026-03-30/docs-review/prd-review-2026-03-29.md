## PRD 内部一致性审查

| # | 优先级 | 问题描述 | 涉及章节 | 修复状态 |
|---|--------|---------|---------|---------|
| P1 | 🟡 中 | **§9.1 视觉样式写"首版提供 2 套成套主题：简洁战术风 / 复盘讲解风"，但 §6.2 球员对象写"球衣卡片式视觉方向参考实况足球 / eFootball 风格球员卡"**。球衣卡片式的 eFootball 风格方向感和"简洁战术风 / 复盘讲解风"两套主题之间的关系没有说清楚。球员展示样式是独立于成套主题的，还是会跟随主题变化？如果独立——OK，但应明确声明两者正交；如果跟随——两套主题各自的球员样式也应列出。 | §9.1 vs §6.2 | ⬜ 未修复 |
| P2 | 🟡 中 | **§6.7 本地保存规则写"本地自动保存优先覆盖当前工作副本，不污染手动命名版本"，但 §13 第一阶段范围和 §14 明确不做都说第一阶段不做历史版本**。既然第一阶段没有"手动命名版本"这个概念，§6.7 提到的"不污染手动命名版本"就是一条悬空规则——它引用了一个第一阶段不存在的实体。建议将 §6.7 的措辞改为"不污染用户已手动保存的项目状态"或类似表述，避免暗示 V1 存在版本系统。 | §6.7 vs §13/§14 | ⬜ 未修复 |
| P3 | 🟡 中 | **§8.1 编辑效率撤销规则写"会改动内容的动作进入撤销栈，例如移动对象、删除对象、修改文本、调整步骤、调整样式设置"，但旧 Requirements 还列了"修改图层、调整权限设置"**。新 PRD 正确地去掉了图层和权限（因为 V1 不做），Requirements §15 也已同步去掉。**这里没有冲突**——只是确认一下两边已对齐。 | §8.1 vs Requirements §15 | ✅ 已对齐 |
| P4 | 🟢 低 | **§6.5 GIF 首发约束写"GIF 首发总时长上限为 15 秒，超出时提示减少步骤数量或降低停留时长"。但"降低停留时长"暗示每帧停留时长是用户可配的，而 §6.5 同时写"GIF 首发只提供 慢 / 标准 / 快 三档速度"**。三档速度是否等同于"每帧停留时长"？如果是——那"降低停留时长"就等于"切换到更快的速度档位"，措辞可以更精确；如果不是——需要明确每帧停留时长和播放速度的区别。 | §6.5 内部 | ⬜ 未修复 |
| P5 | 🟢 低 | **§7.5 历史版本（后续阶段）详细定义了自动快照规则（10 分钟、最多 5 个等），但 §16 关键产品模型没有在"可能的后续扩展实体"中列出 AutoSnapshot**。§16 只列了 `ProjectVersion`。这不算冲突，但如果后续要实现版本系统，AutoSnapshot 和 ManualVersion 是否作为 ProjectVersion 的子类型应明确。 | §7.5 vs §16 | ⬜ 未修复 |
| P6 | 🟢 低 | **§10.2 安装与分发写"Windows 双击项目文件直接打开、Android 从聊天软件/文件管理器直接唤起应用，作为后续增强能力评估"，但 §6.7 和 §13 都明确第一阶段不做项目文件导入导出**。这意味着"双击项目文件打开"在第一阶段没有文件格式可双击。§10.2 的后续增强表述本身不矛盾，但它的**前提依赖**（项目文件格式）应明确标注。 | §10.2 vs §6.7/§13 | ⬜ 未修复 |
| P7 | 🟢 低 | **§9.2 跨端体验写"平板设备按 Android 端交互策略适配"，但没有明确平板是否支持横屏编辑**。PRD §5 产品原则写了"Android 端优先保证触控场景下的新建、编辑、保存、导出完整可用"，这个"完整可用"是否包含横屏大屏幕编辑体验？平板 vs 手机的差异可以独立标注。 | §9.2 vs §5 | ⬜ 未修复 |

**最值得优先修复的 4 项：**

1. **P1** — 球员展示样式 vs 成套主题的关系未定义（正交还是跟随？）
2. **P2** — §6.7 引用了第一阶段不存在的"手动命名版本"概念
3. **R1** — Requirements 比 PRD 多了两条重要保存规则，PRD 应吸收
4. **R2** — PRD 的同名文件导入策略应同步到 Requirements

**新 PRD 整体质量很高**——产品方向清晰、边界明确、后续扩展有预留。以上问题修完即可冻结作为开发基准。
