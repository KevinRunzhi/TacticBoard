# GitHub 同类项目调研与借鉴清单

更新日期：2026-03-26

## 1. 目的

这份文档用于整理 GitHub 上与“足球战术板 / 阵容编辑 / 战术动画 / 分享出图”相关的开源项目，并回答三个问题：

- 有没有真正接近本项目目标形态的开源仓库
- 这些项目里哪些内容值得学习或借鉴
- 哪些实现方式或产品取舍不适合直接照搬

本文主要关注公开 GitHub 仓库，不包含商业 SaaS 产品本身。

## 2. 总体判断

结论很明确：

- GitHub 上有一些“相近方向”的项目，但大多数只覆盖其中一层能力。
- 常见形态是：
  - 简单阵容编辑器
  - 战术板原型或 demo
  - 面向数据动画的可视化工具
- 目前没有发现一个开源仓库同时覆盖本项目文档里已经明确的这些能力：
  - 专业编辑器
  - 多步骤帧 / 残影 / 演示
  - 项目 / 模板 / 球队 / 阵容预设 / 球员资料库
  - 分享页与副本二创
  - PNG / GIF / MP4 输出
  - 个人空间 / 团队空间 / 轻协作权限

这意味着：

- 不能把任何一个现成仓库直接当成“整体蓝本”
- 但可以把它们拆开，当作不同层面的参考样本

## 4. 单项目拆解

### 4.1 gljubojevic/tactics-board

仓库链接：
- [gljubojevic/tactics-board](https://github.com/gljubojevic/tactics-board)

我看到的关键信息：

- 仓库定位是“五人制战术板”
- 技术栈是 `React + Material UI + Firebase`
- README 明确写了编辑与出图基础是 `SVG`
- 保存思路是“可编辑 JSON 定义”与“PNG 导出”分离
- 动画模型是 `key frames`
- 分享与协作目标里明确出现了 `view only / edit / collaborate`

从仓库描述可直接确认的点：

- “Base for editor and image creation is SVG”
- 图形通过修改 shape 参数后重新生成 SVG
- 新动画帧通过复制上一帧创建
- 物体移动路径可通过 b-spline 编辑
- 文档保存与分享是这个项目的重要方向

最值得学习的内容：

- **编辑器数据模型方向**
  - 这个项目明显是在把画布对象当作结构化图元处理，而不是把整个画布当成一次性位图。
  - 对本项目最有价值的是这个方向本身：球员、球、线、区域、文本都应该有独立对象身份。

- **步骤 / 帧模型**
  - 它的关键帧设计和你文档里的“复制上一帧继续编辑”高度相近。
  - 这说明你的“步骤帧 + 残影 + 播放”路线是合理的，不是凭空设计出来的。

- **SVG 适合高质量出图**
  - 这个仓库把 SVG 作为编辑与导出基础，说明它也在优先考虑“图像表达质量”和“对象级控制”。
  - 对你的项目尤其重要，因为你强调社媒出图、导出和风格一致性。

- **分享与存储天然适合对象化结构**
  - 如果保存的是对象 JSON，而不是最终位图，那么分享页、复制副本、历史版本、模板化都会更顺。

不建议直接照搬的地方：

- 它更像“参考某现成站点功能的原型实现”，不是完整产品架构。
- 它的 README 重点在编辑器和图元，没有延展到你当前文档里最重要的资产体系和页面架构。
- 它提到协作，但没有体现成熟的团队空间、权限模型和产品层边界。

对本项目的具体启发：

- 如果你的第一版要优先保证：
  - 出图质量
  - 步骤帧能力
  - 分享页一致性
  - 图层和对象的可控渲染
  那么 `SVG + 对象模型 + 帧间身份映射` 很值得作为主路线优先评估。

### 4.2 zenggo/soccer-tactical

仓库链接：
- [zenggo/soccer-tactical](https://github.com/zenggo/soccer-tactical)

我看到的关键信息：

- README 非常短，定位是 “A football tactical board run on PC browser”
- 仓库 topics 标注了 `html5`、`canvas`
- 代码量很小，提交历史也很少
- 整体更像轻量 demo 或最小可运行原型

最值得学习的内容：

- **最小战术板 MVP 的收敛方式**
  - 这个项目说明：只做浏览器端战术板，不先做复杂后端和资产系统，也是一个可验证的起点。
  - 如果你未来想单独拆一个“编辑器交互实验仓库”，这种收敛方式是对的。

- **Canvas 路线适合先验证手感**
  - 对象拖拽、线条绘制、基础 UI 响应，Canvas 可以非常快地搭起来。
  - 如果目标是快速验证“球员拖拽是否顺手”“线路绘制手感是否自然”，这种路线有价值。

- **技术复杂度低**
  - 对前期实验来说，少依赖、少结构、少产品层抽象，能更快得到第一轮交互反馈。

不建议直接照搬的地方：

- 它只强调 PC 浏览器，没有多端策略。
- 看不出成熟的模板、分享、版本、资产、团队空间设计。
- 这种“轻量 tactical board”适合 demo，不适合作为你当前产品的完整骨架。

对本项目的具体启发：

- 可以借它的思路做一个“编辑器内核验证版”。
- 但不能让整个项目停留在这种层级，否则会偏离你已经收敛好的产品方向。

### 4.3 spyderkam/Tactics-Board

仓库链接：
- [spyderkam/Tactics-Board](https://github.com/spyderkam/Tactics-Board)

我看到的关键信息：

- README 标题直接写了“real-time interactive football tactics board”
- 特性包括：
  - 拖拽球员
  - 阵型预设
  - 可移动足球
  - 线路绘制
  - 实时协作
- 技术栈写得很清楚：
  - `Flask + SocketIO`
  - `HTML5 Canvas`
- README 还明确列出了已知问题：
  - 拖拽性能
  - 显示号码后的性能显著变差
  - 移动端体验需要加强
  - 形状工具还需要打磨

最值得学习的内容：

- **实时协作的最小实现路径**
  - 这个项目说明，多人协作未必需要一开始就上复杂的 CRDT 或 OT。
  - 对足球战术板这种对象数可控、同步频率也可控的场景，`WebSocket + 事件广播` 完全可以先做出第一版。

- **把协作当成后加层，不要绑死在编辑器内核里**
  - 它的前后端拆法提示了一个很实用的方向：
  - 先把本地编辑事件定义清楚，再决定哪些事件需要广播。

- **README 里的“已知问题”本身就是很好的反向教材**
  - 拖拽性能差，往往意味着渲染和对象管理没有很好分层。
  - 显示号码就明显卡顿，说明文本渲染和重绘成本需要谨慎设计。
  - 移动端难做，说明桌面端交互若没有提前抽象，后面补手机端会很痛。

不建议直接照搬的地方：

- 这个项目现在更像 beta 实验品，而不是产品级实现。
- Canvas 方案在对象越来越多、文字越来越多、图层和导出要求越来越高时，复杂度会上升得很快。
- 你的第一阶段文档已经明确不做实时协作，所以不能让“多人同步”牵着架构走。

对本项目的具体启发：

- 第一阶段不做实时协作是对的。
- 如果以后要加协作，完全可以先做：
  - 对象级编辑事件
  - 房间级广播
  - 低频保存
  - 冲突尽量通过产品限制规避
- 没必要一开始就把编辑器架构绑在协作系统上。

### 4.4 remidej/11-builder

仓库链接：
- [remidej/11-builder](https://github.com/remidej/11-builder)

我看到的关键信息：

- README 定位是：
  - “create your own football lineup”
  - “customize its look”
  - “download it as a ready-to-share image”
- 技术栈是 `React`
- 仓库还用了 `Cheerio` 去抓取球员图片和信息来源
- 作者在贡献建议里提到：
  - 分享到外部平台
  - 开放 API 提供球员图片和信息

最值得学习的内容：

- **内容成品感优先**
  - 这个项目的思路不是“做个后台”，而是“做出一张能直接分享的阵容图”。
  - 这和你文档里“内容创作友好”“社媒出图”“好看出图”的方向是对齐的。

- **数据和视觉可以结合**
  - 它不是只放 11 个圆点，而是尝试结合球员图片和现实数据源。
  - 这提醒你：未来即使不接真实球员数据库，球员头像、球队视觉、样式模板也很值得做强。

- **分享图产品感**
  - 对你来说，分享出去的内容不是“编辑器截图”，而应该是“完成品”。
  - 这个项目值得借的就是这层意识。

不建议直接照搬的地方：

- 它本质上是阵容图工具，不是战术步骤编辑器。
- 它不强调步骤、线路、区域、分享页观看、项目版本和资产沉淀。
- 如果照它的产品边界走，会把你的项目收缩成一个 lineup maker。

对本项目的具体启发：

- 你的分享图导出和模板风格，应该以“成片感”而不是“工具感”为目标。
- “球衣卡片式球员样式”“头像”“颜色与品牌化输出”都值得继续保留。

### 4.5 jalapic / soccermatics tactics board 系列 gist

  - 球场
  - 球员点位
  - 基础拖拽
  - 少量绘制增强

  - 如果只是想验证“球场 + 球员 + 球 + 拖拽”这一层，它们非常直接。

- **低门槛验证交互手感**
  - 很适合在一两天内快速试试：
  - 站位摆放
  - 拖拽阻尼
  - 对象吸附
  - 简单工具条交互

不建议直接照搬的地方：

- 太早期了，无法承接真正的产品层能力。
- 对你现在的目标来说，这类项目最多只能参考“最小交互实验”，不能参考“产品结构”。

### 4.6 Dato-Futbol/soccerAnimate

仓库链接：
- [Dato-Futbol/soccerAnimate](https://github.com/Dato-Futbol/soccerAnimate)

我看到的关键信息：

- 它是一个 R 包，不是战术板编辑器
- 核心方向是：
  - 2D 足球 tracking data 动画
  - 静态图
  - GIF 动画
  - 球队与球员层面的分析视图

最值得学习的内容：

- **动画和输出是独立能力**
  - 它提醒了一点：动画生成并不一定要完全绑定在编辑器操作层。
  - 输出层可以有自己的一套渲染和参数配置逻辑。

- **可视化表达的丰富度**
  - 它做了多种表现方式，比如不同分析层、不同高亮方式。
  - 对你的项目来说，这更像“未来内容输出增强”的灵感来源。

不建议直接照搬的地方：

- 它是数据可视化，不是创作型产品。
- 交互、资产、页面、协作、模板、分享页这些都不是它的重点。

## 5. 可以直接吸收的借鉴点

下面这些点，适合直接进入本项目后续技术设计或产品设计讨论：

### 5.1 编辑器内核

- 采用“对象化建模”而不是位图思路
- 让球员、球、线、文本、区域都有稳定对象 ID
- 步骤帧通过“复制上一帧 + 增量修改”建立
- 帧间通过对象身份映射支撑残影、差异对比、单人视图等能力

最主要参考：
- `gljubojevic/tactics-board`

### 5.2 动画与步骤系统

- “关键帧 / 步骤帧”是足球战术讲解里可行的主模型
- 动画 UI 不必复杂，先从：
  - 播放
  - 暂停
  - 上一步 / 下一步
  - 时间轴或缩略帧
  做起

最主要参考：
- `gljubojevic/tactics-board`

### 5.3 分享与导出思路

- “可编辑源数据”与“最终导出结果”应该是两套概念
- 分享页最好基于结构化项目快照，而不是编辑器截图
- 导出时应优先考虑成品感，而不是单纯“导一张图”

最主要参考：
- `gljubojevic/tactics-board`
- `remidej/11-builder`

### 5.4 协作的后续扩展路径

- 实时协作不应先行
- 如果以后做，可以先从对象级事件同步开始
- 不必一开始就设计复杂多人并发模型

最主要参考：
- `spyderkam/Tactics-Board`

### 5.5 MVP 节奏控制

- 可以先单独验证编辑器手感，再叠资产体系
- 但正式产品不能永远停留在 demo 级别

最主要参考：
- `zenggo/soccer-tactical`

## 6. 明确不要照搬的坑

这些是从调研里反过来得到的警示：

- **不要只做成“能拖球员”的 demo**
  - 这是 GitHub 上最常见的层级，也是最不稀缺的层级。

- **不要把分享结果做成“编辑器截图感”**
  - 你的差异点之一就是内容输出质量。

- **不要过早把实时协作放进第一阶段**
  - 会显著增加复杂度，还容易拖慢编辑器内核的收敛。

- **不要在未验证数据模型前把所有能力都绑死在 Canvas 重绘逻辑里**
  - 一旦后面要加图层、导出、版本、分享、步骤信息、文字样式，成本会迅速上升。

- **不要误把 lineup builder 当成完整战术板**
  - 阵容图产品和战术编辑器有重叠，但不是同一个问题。

## 7. 对本项目的建议结论

结合当前文档和这轮调研，我认为更合理的方向是：

- **产品层**
  - 继续维持你当前的页面与资产结构，不要因为 GitHub 上现成项目较轻，就把产品收缩成单页画板。

- **编辑器层**
  - 优先做专业对象模型和步骤帧模型。
  - 在技术路线评估上，`SVG 主导` 比 `纯 Canvas 主导` 更值得优先审视。

- **分享与导出层**
  - 把“出图质量”和“可传播性”当成核心能力，而不是附属按钮。

- **协作层**
  - 第一阶段维持非实时协作判断。
  - 后续如需扩展，再走事件同步路线。

## 8. 推荐的后续动作

如果把这轮调研转成可执行动作，建议顺序如下：

1. 先做一份“编辑器内核技术选型比较”
   - `SVG vs Canvas vs 混合方案`
   - 重点比较对象建模、步骤帧、导出、性能、移动端适配

2. 再做一份“战术板数据模型草案”
   - Project
   - Step
   - Object
   - Layer
   - Template
   - Team / Lineup / PlayerProfile
   - ShareSnapshot

3. 最后再进入页面级 UI 与实现拆解
