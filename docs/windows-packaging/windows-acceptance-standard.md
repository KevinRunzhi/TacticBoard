# Windows 打包验收标准

## 1. 文档目标

这份文档定义 TacticBoard 在 Windows 打包第一阶段的专项验收标准。

目标：

- 让 Windows 打包验收从“是否能启动”提升到“是否可交付”
- 统一功能验收、平台验收和失败场景验收标准
- 给后续实现和手动测试提供明确清单

## 2. 验收范围

本文件覆盖：

- Tauri 基础接入
- Windows 路由稳定性
- 本地保存与继续编辑
- PNG / GIF 原生保存
- 参考底图原生导入
- Windows 产物可启动性

本文件不覆盖：

- Android 打包
- 自动更新
- 代码签名
- 商店发布

## 3. 验收等级

### 3.1 可开发验收

满足：

- `tauri:dev` 能启动
- 主链路能跑通
- 主要接口行为正确

### 3.2 可内部演示

满足：

- 打包产物可运行
- 主链路稳定
- PNG / GIF / 参考底图链路稳定
- 主要失败分支有合理提示

### 3.3 可交付展示

满足：

- Windows `x64` 产物稳定
- 已有明确安装与运行说明
- 已明确未签名产物的安全提示预期

## 4. 环境验收

必须通过：

- 本机 Node.js 环境正常
- 本机 Rust toolchain 正常
- 本机 C++ Build Tools 正常
- 本机 WebView2 Runtime 正常

通过标准：

- `npm run tauri:dev` 不因环境缺项失败
- `npm run tauri:build` 不因工具链缺项失败

## 5. 配置验收

必须通过：

- `tauri.conf.json` 已配置：
  - `beforeDevCommand`
  - `beforeBuildCommand`
  - `devUrl`
  - `frontendDist`
- `devUrl` 与 Vite `8080` 端口一致
- `vite.config.ts` 已具备：
  - 固定端口
  - `strictPort: true`
  - Tauri 开发宿主适配

## 6. Web 回归验收

Windows 打包接入不能以破坏现有 Web 版本为代价。

必须通过：

- `npm run dev` 仍可正常启动
- 浏览器环境下工作台、项目页、设置页、编辑器仍可访问
- 浏览器环境下主链路 `工作台 -> 新建/打开 -> 编辑 -> 保存 -> 返回 -> 继续编辑` 仍可走通
- 浏览器环境下 PNG / GIF 导出仍可工作
- 浏览器环境下参考底图导入仍可工作

失败判定：

- 为了接入 Tauri 而导致 Web 路由失效
- 为了接入原生保存而破坏浏览器导出
- 为了接入原生文件选择而破坏浏览器 `File` 导入路径

## 7. 路由验收

必须通过：

- Windows 桌面端不再依赖单一浏览器路由策略
- 工作台可打开
- 项目页可打开
- 设置页可打开
- 编辑器可打开
- 刷新当前页面不会进入空白页
- 刷新当前页面不会进入错误 404

## 8. 主链路验收

必须通过以下完整路径：

1. 打开工作台
2. 新建项目
3. 进入编辑器
4. 添加或编辑对象
5. 保存正式项目
6. 返回工作台
7. 再次打开项目
8. 内容仍存在

失败判定：

- 保存后项目消失
- 重开后项目内容明显丢失
- 返回工作台后主链路中断

## 9. 编辑器核心操作验收

必须通过：

- 球员新增、编辑、删除
- 文本新增、编辑、删除
- 线路新增、编辑、删除
- 区域新增、编辑、删除
- 步骤新增、复制、删除、重排

要求：

- Windows 打包不应破坏现有编辑器主功能

## 10. 文件保存验收

### 9.1 PNG

必须通过：

- 导出 PNG 时可选择路径
- 文件成功写入本地
- 文件可正常打开
- 取消保存时不报错
- 保存失败时有明确提示

### 9.2 GIF

必须通过：

- 导出 GIF 时可选择路径
- 文件成功写入本地
- 文件可正常播放
- 取消保存时不报错
- 保存失败时有明确提示

### 9.3 统一要求

- Windows 端导出不能依赖浏览器下载行为
- 导出失败不能破坏当前编辑状态

## 11. 参考底图导入验收

必须通过：

- 可打开原生文件选择器
- 可选择图片并导入
- 导入后画布可显示参考底图
- 取消选择时不报错
- 图片过大时有提示
- 图片不合法时有提示

失败判定：

- 只能打开对话框但无法真正导入
- 组件仍依赖浏览器 `input[type=file]`

## 12. 存储验收

必须通过：

- 正式 Windows 产物关闭后重开，项目仍可读取
- 同一正式 Windows 产物中，本地保存和继续编辑可用

必须明确：

- 浏览器版数据不承诺自动迁移到桌面版
- `tauri dev` 数据不承诺自动迁移到正式安装包

这两项在第一阶段不是 bug。

## 13. 构建产物验收

必须通过：

- `npm run tauri:build` 成功
- 生成 Windows `x64` 产物
- 至少一种主要产物类型可分发

其中“构建通过”和“本机双击产物可启动”必须分开记录：

- `tauri:build` 成功只说明构建链路和打包链路通过
- 产物实际双击启动属于手动 OS 级验证，未执行时不能写成已通过
- 验收记录优先写明产物输出目录，例如 `src-tauri\\target\\release\\bundle\\nsis\\`
- 不建议把某一个固定安装包文件名写成唯一验收证据，因为它可能随产品名或环境编码表现变化

必须说明：

- 如产物未签名，SmartScreen 提示属于预期现象

## 14. 失败场景验收

必须覆盖：

- 用户取消导出
- 用户取消选择图片
- 导出保存失败
- 图片读取失败
- 图片过大
- 当前路由刷新

要求：

- 用户取消不应显示错误
- 失败提示要能区分原因
- 失败后仍可继续使用应用

## 15. 推荐验收命令

建议顺序：

```bash
npm run build
npm run test
npm run lint
npm run tauri:dev
npm run tauri:build
```

## 16. 手动验收场景

### 场景 A：桌面启动

- 启动桌面应用
- 进入工作台

### 场景 B：主链路

- 新建项目
- 编辑
- 保存
- 返回工作台
- 再打开

### 场景 C：PNG 导出

- 导出 PNG
- 选择路径
- 检查文件

### 场景 D：GIF 导出

- 导出 GIF
- 选择路径
- 检查文件

### 场景 E：参考底图导入

- 选择图片
- 导入成功
- 关闭重开后检查项目

### 场景 F：失败分支

- 取消保存
- 取消选择图片
- 使用超大图片

## 17. 验收结论模板

实施完成后，验收结论建议至少包含：

- 是否通过 `tauri:dev`
- 是否通过 `tauri:build`
- 是否仅验证到“产物已生成”，还是已经完成“本机双击启动”
- 是否通过主链路
- 是否通过 PNG 保存
- 是否通过 GIF 保存
- 是否通过参考底图导入
- 还存在哪些已知限制

## 18. 当前结论

Windows 打包第一阶段验收重点不是“能不能跑起来”，而是：

- 能不能稳定启动
- 能不能稳定保存
- 能不能稳定导出
- 能不能稳定继续编辑

达到这些，才算真正完成第一阶段。
