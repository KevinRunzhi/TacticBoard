# 06-quality

这一层负责定义质量门槛、验收方式和已知风险。

当前文档：

- `acceptance-checklist.md`
- `test-matrix.md`
- `known-risks.md`

使用规则：

- 这一层回答“做完怎么算完成、要测什么、最容易在哪里翻车”
- 中型和大型代码改动的默认质量门槛，也以这一层为准，而不是只看单次口头约定。

## 当前质量执行原则

- Validation is part of the task, not an optional follow-up.
- 对中型和大型代码改动，执行者必须先自己完成验证，再交付结果；不能只在结尾建议别人去跑命令。
- 验证深度必须与风险、影响面和用户影响匹配；不能用单个浅层检查替代整体验收。
- 涉及保存、导出、导入、路由、恢复、平台桥、打包配置时，不能只测 happy path。
- 如果相关自动化测试存在，就必须运行；如果关键行为缺测试且补测试可行，应优先补测试。
- 如果本轮无法运行某个必要命令或必要人工场景，必须明确写出阻塞点、未验证项和剩余风险，不能直接宣称完成。

## 默认改动分级

- 小改动：
  - 单点文案、样式、注释、或局部非行为性修正
  - 不改变主链路、状态语义、平台边界或配置
- 中型改动：
  - 跨多个文件的用户可见行为修改
  - 触及路由、保存、导出、导入、恢复、共享状态、平台桥
  - 同时改运行时代码和测试，或同时改代码与配置
- 大型改动：
  - 跨多个层级或多个主功能
  - 触及持久化规则、平台契约、运行时身份、打包配置
  - 需要验证多个主流程和失败分支，才能判断结果可信

如果拿不准，按更高一级处理。

## 中型 / 大型改动最低要求

- 必须由执行者自己实际运行相关验证命令。
- 必须在 `docs/DocsReview/` 留下 dated 记录。
- 至少记录：
  - 范围
  - 改动级别
  - 实际运行的命令
  - 实际跑过的人工场景
  - 未验证项
  - 剩余风险
- 不能把“建议运行”“理论上应当通过”“之前某轮通过过”当作本轮验证证据。
