# V1 验收标准

最后更新：2026-04-23

## 1. 目的

本标准用于界定足球战术板项目当前版本是否达到 `V1 可演示`。

判定重点不是“功能是否齐全”，而是以下 4 点是否成立：

- 主链路可以稳定走通
- 核心编辑操作可以完成
- 导出结果可用
- 异常路径不会在演示过程中中断流程

## 2. 验收等级

### 2.1 可内部验收

满足以下条件：

- 工作台、项目页、编辑器、设置页可正常打开
- 本地保存、返回工作台、重新打开继续编辑可正常完成
- 至少 1 条完整编辑链路和 1 条完整导出链路通过
- 已知问题不会直接导致白屏、崩溃或数据丢失

### 2.2 可对外演示

在“可内部验收”的基础上，还应满足：

- 编辑器核心对象操作可连续演示
- 主要边界情况有清晰降级或提示
- PNG 和 GIF 导出结果经过人工核对
- 无明显开发态占位、失效按钮或错误文案

## 2.3 证据等级规则

- “可开发验证”或“能跑起来”不等于“可内部验收”。
- “构建成功”不等于“行为已通过”。
- “之前某一轮通过过”不等于“本轮改动已通过”。
- 对中型和大型代码改动，本轮验收结论必须基于执行者本轮亲自运行的命令和人工场景。
- 如果某个必要验证本轮没跑，就必须在结论中明确写成“未验证”，不能默认算通过。

## 3. 验收范围

### 3.1 页面与入口

- 工作台
- 项目页
- 编辑器
- 设置页
- 404 页面

### 3.2 主链路

- 工作台 -> 新建项目 -> 编辑器 -> 本地保存 -> 返回工作台 -> 继续编辑
- 工作台 -> 打开最近项目 -> 编辑器 -> 保存 -> 返回工作台
- 编辑器 -> PNG 导出
- 编辑器 -> GIF 导出

### 3.3 核心编辑对象

- Player
- Ball
- Text
- Line
- Area
- Step
- Match Meta
- Reference Image

## 4. 验收项

### 4.1 启动与入口

- 应用可以正常启动，无白屏和崩溃
- 工作台可以正常渲染
- `新建项目` 入口可用
- `继续上次编辑` 在有草稿时可用，无草稿时提示正确
- `最近项目` 可正常显示
- 项目页可正常打开
- 设置页可正常打开
- 无登录、注册、分享、团队等旧入口残留

### 4.2 主链路

- 从工作台点击 `新建项目` 可以进入编辑器
- `mode=new` 不会错误恢复旧草稿
- 编辑器修改后自动保存草稿生效
- 首次手动保存后生成正式项目
- 保存后返回工作台可以看到该项目
- 从工作台重新打开项目后内容与保存前一致
- 删除项目后工作台和项目页状态刷新正确

### 4.3 编辑器核心操作

- 可以添加、移动、编辑、删除球员
- 可以移动足球
- 可以添加、移动、编辑、删除文本
- 可以添加、编辑、删除线路
- 可以添加、编辑、删除区域
- 可以编辑项目名称
- 可以编辑步骤说明
- 可以编辑比赛信息

### 4.4 步骤管理

- 可以新增步骤
- 可以复制步骤
- 可以删除步骤
- 至少保留 1 个步骤
- 可以左右重排步骤
- 切换步骤后内容正确
- 复制步骤后内容独立，不发生串改
- 删除和重排步骤后当前选中逻辑正确

### 4.5 参考底图

- 可以导入参考底图
- 可以调整显示或隐藏
- 可以调整缩放和平移
- 可以移除参考底图
- 项目重新打开后参考底图仍可恢复

### 4.6 导出

- PNG 导出成功
- PNG 文件可正常打开
- PNG 结果与当前画布一致
- 标题导出开关有效
- 比赛信息导出开关有效
- 参考底图导出开关有效
- GIF 导出成功
- GIF 文件可正常播放
- GIF 帧顺序正确
- GIF 速度档位有效
- GIF 超限时有明确提示

### 4.7 异常与边界

- 空项目状态不崩溃
- 草稿不存在时继续编辑不崩溃
- 项目不存在时进入编辑器有合理降级
- 清除本地数据后页面状态正确
- 导入非法图片或失败时有提示
- 导出失败时有提示
- 删除操作不会无反馈误删
- 制式切换三选一流程正确

### 4.8 移动端与小屏

- 小屏下可以进入编辑器
- 小屏下可以保存项目
- 小屏下可以打开最近项目
- 小屏下属性抽屉可用
- 小屏下步骤抽屉可用
- 小屏下导出入口可用

### 4.9 演示观感

- 没有明显开发态占位文案
- 没有明显失效按钮
- 当前工具高亮清楚
- 当前选中对象反馈清楚
- 保存状态文案一致
- 返回工作台后的提示清楚

## 5. 验收方法

### 5.1 自动化验证

- `npm run build`
- `npm run test`
- `npm run lint`
- `git diff --check`

对中型和大型代码改动：

- 上述命令是最低自动化门槛，不是可选项。
- 如果改动触及 Tauri、保存、导出、导入、路由运行时或平台桥，还应补桌面侧验证。

### 5.2 人工验收

人工验收至少覆盖以下演示路径：

1. 新建空白项目并保存
2. 打开已有项目并继续编辑
3. 创建至少 2 个步骤并导出 PNG
4. 创建至少 2 个步骤并导出 GIF
5. 导入参考底图并验证导出开关

对中型和大型代码改动：

- 必须由执行者本人完成与本轮改动面直接相关的人工验证。
- 不能只测 happy path；保存、导出、导入、恢复、路由、平台边界至少要覆盖一组失败 / 取消 / 异常分支。
- 如果本轮没有完成必要人工验证，不得宣称达到“可内部验收”。

### 5.3 结果记录

每一轮验收都应在 `docs/DocsReview` 下形成单独报告，至少包含：

- 本轮范围
- 改动级别
- 已通过项
- 未通过项
- 修复项
- 实际执行的命令
- 实际执行的人工场景
- 未验证项
- 剩余风险
- 是否达到 `可内部验收` 或 `可对外演示`

## 6. 当前 V1 目标线

本项目当前阶段的验收目标是：

- 达到 `可内部验收`
- 向 `可对外演示` 收敛

当前阶段不要求：

- 图层系统
- 批量操作
- 拖拽式步骤时间轴
- 线路端点拖拽
- 完整模板系统
- 桌面壳和 Android 壳打包
